# v0.3.0 — Infrastructure & Hardware Monitoring Expansion

**Branch:** `feature/v0.3.0-infrastructure-hardware-monitoring`
**Status:** in progress

## Goal

Expand the Infrastructure page from a single "container-visible system metrics"
widget into a polished infrastructure & hardware monitoring centre: storage
overview, drive inventory (temps / SMART), pool health, hardware metric cards
(CPU, memory, GPU, network, storage, uptime) with detail views, an alerts
summary, and clear telemetry source status — all degrading gracefully when data
is unavailable.

## Guiding constraints (from brief + repo invariants)

- **No DB schema changes.** Telemetry is read live, server-side, per request. We
  reuse the existing `metrics` table only for the already-persisted rate values
  (CPU %, aggregate net rate) via `getLatestMetrics()`; everything else is fresh.
- **No external integrations.** TrueNAS/SMART/ZFS remain v0.4+. We read only
  **read-only, local, container-visible** sources (`/proc`, `/sys`) plus the
  already-opt-in Docker socket for container-level stats. Sources that need
  privileged tooling (smartctl, zpool, nvidia-smi) degrade to a clean
  `unavailable` / `not_configured` state — never a fault.
- **No shell execution.** Pure file reads under `/proc` and `/sys` only.
- **Server-side only + redaction.** No secrets, socket paths, host FS paths,
  serials, or raw command lines reach the client. Disk serials are redacted.
- **Keep the current dashboard the default.** The home `SystemOverviewWidget`
  is untouched; this is purely additive on the Infrastructure route.

## Severity model

`HealthSeverity = "healthy" | "warning" | "critical" | "unavailable"`

- Temperature: `<45 healthy`, `45–54 warning`, `>=55 critical`.
- Capacity: `<warnThreshold healthy`, `warn–critical warning`, `>=critical`.
- `unavailable`/unknown is **never** critical absent real failure evidence.

`TelemetrySourceStatus = "available" | "partial" | "unavailable" | "not_configured"`

## Files

### Server — telemetry abstraction (`src/server/telemetry/`)
- `proc.ts` — pure parsers + guarded readers for `/proc/meminfo`, `/proc/stat`,
  `/proc/net/dev`, `/proc/mounts`.
- `sysfs.ts` — drive inventory (`/sys/block/*`), hwmon temps/power, thermal
  zones, best-effort GPU (amdgpu hwmon).
- `redact.ts` — serial/path redaction (pure, tested).
- `severity.ts` — temp/capacity/memory-pressure classification (pure, tested).
- `container-stats.ts` — best-effort per-container CPU/mem/net via the existing
  Docker engine client; bounded + degrades to unavailable.
- `service.ts` — orchestrator → builds `InfrastructureTelemetryDTO`, computes
  source statuses + alerts, applies redaction.
