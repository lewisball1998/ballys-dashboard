# v0.2.7 — Icon Pack Research & Alias Expansion (Consolidated Research Report)

**Status:** Research / planning only — not implemented. No code, schema, Docker, or
auth changes were made. Produced by a parallel 7-agent research pass (6 research
agents + consolidation).

**Current released version:** v0.2.6 — App Icon Library & Custom Icons
**Target phase:** v0.2.7 — Icon Pack Research & Alias Expansion
**Suggested implementation branch:** `feature/v0.2.7-icon-alias-expansion`

---

## Executive summary

The v0.2.6 foundation (typed `apps.icon` refs, pure `resolveIconSrc`/`suggestIconKey`,
mask-rendered first-party monochrome glyphs, PNG/WebP custom uploads, SVG blocked) is
the right base and needs **no architectural change** to grow coverage. The biggest UX
gap is **suggestion coverage**: hundreds of common homelab apps currently fall back to
`generic` or get mis-mapped, because the registry has only 16 keys and sparse aliases.

Two complementary, low-risk moves close most of the gap:

- **(A) large alias expansion** of existing keys, and
- **(B) a curated batch of ~11–14 new first-party generic keys** (notably `ai`, `proxy`,
  `identity`, `automation`, `camera`, `backup`, `notes`, `finance`, `rss`, `server`,
  `indexer`).

Both are code+asset changes only — **no DB migration, no Docker/auth changes**.

On packs/licensing, the verdict is decisive: **copyright license ≠ trademark license.**
Generic glyph sets (Lucide/Tabler/MDI) are safe to bundle; brand-logo packs (selfh.st,
homarr-labs/dashboard-icons, simple-icons) are permissively *copyrighted* but
**trademark-risky to bundle** in a publicly distributed product — they should be
**user-import-only**. The safe path is to keep shipping zero third-party brand assets
and, in a *future* phase, add an opt-in user-driven icon-pack import (JSON manifest,
raster-only, magic-byte validated, `pack:<id>/<key>` grammar). **That import feature is
explicitly NOT part of v0.2.7.**

---

## Recommended v0.2.7 scope

**Option A (alias expansion) + a curated slice of Option B (larger first-party generic
set). Defer C/D/E (import, bundled third-party packs, brand metadata) to a later phase,
captured now as an ADR.**

Rationale: A+B are zero-migration, backward-compatible, fully within the v0.2.6 model,
and deliver the most user value (good auto-suggestions for the apps people actually run).
Import/licensing is the *research* deliverable that de-risks the future; building it now
would over-scope a point release and re-open SVG/SSRF/trademark surfaces the constraints
forbid.

Concrete v0.2.7 deliverables:

1. Add **new first-party generic keys** (Tier 1 below; optionally Tier 2) — each with an
   authored monochrome SVG glyph.
2. **Expand aliases** on existing + new keys to cover the catalogued P1/P2 apps.
3. Add a **qualifier-stripping pre-pass** to `suggestIconKey` for 4K/HD/multi-instance
   names.
4. **Unit tests** mapping the catalogued apps → expected keys; tests for the strip
   pre-pass and short-alias non-bleed.
5. **ADR** documenting the icon policy + the future icon-pack import design (extends
   ADR 0013).

## Suggested additions to the existing generic icon registry

**Recommended NEW first-party generic keys (authored monochrome glyphs, NO brand logos):**

- **Tier 1 — strongly recommended (11):** `ai`, `proxy`, `identity`, `automation`,
  `camera`, `backup`, `notes`, `finance`, `rss`, `server`, `indexer`. (Each fills a large
  cluster with NO good existing home; biggest single win is `ai`.)
- **Tier 2 — include if appetite (3):** `calendar`, `search`, `requests`.
- **Defer/fold:** `vpn` (keep as `shield` alias), `network-node` (→`monitor`/`chart`),
  `bell` (→`mail`), `tasks` (→`generic`), `sync` (→`cloud`), `document` (→`notes`),
  `bookmark` (→`book`), `clock` (→`monitor`), `bulb` (defer).

This takes the registry from **16 → ~27–30 keys** — still a curated, cohesive first-party
set, not a top-200 pack.

**Glyph notes (generic, currentColor-mask friendly):** `ai`=chip-with-nodes/neural dots;
`proxy`=routing fork/gateway; `identity`=key (or key+person); `automation`=connected
flow-nodes or gears; `camera`=CCTV body+lens; `backup`=disk + circular restore arrow;
`notes`=page+lines (+pencil); `finance`=wallet/coin (locale-neutral, no `$`);
`rss`=standard broadcast arcs; `server`=rack/tower with slats; `indexer`=magnifier over
stacked lines; `calendar`=grid+rings; `search`=magnifier; `requests`=list/inbox with `+`.

**Alias expansion (apply to existing keys; dedup against current arrays):** large
additions to `media`, `download`, `shield`, `container`, `chart`, `monitor`, `database`,
`code`, `cloud`, `storage`, `book`, `home` (full token lists are in the four app-agent
outputs). Guidance: **all aliases normalized lowercase-alphanumeric; avoid aliases
< 3 chars** (they only exact-match and risk cross-bleed — prefer long-form, e.g.
`nginxproxymanager` not `npm`, `wireguard` not `wg`).

## Suggested 4K / multi-instance alias handling

- **Primary mechanism: a qualifier-stripping pre-pass in `suggestIconKey`** (before
  scoring). Strip known qualifier tokens — `4k, uhd, hd, sd, 1080p, 2160p, fhd, anime,
  kids, remux, x265, hevc, instance, main, alt, secondary, 2, 3` — then match the
  residual. **Guard: never strip to empty** — if removal empties the string, keep the
  original (so bare `Anime`/`Movies`/`Music`/`Photos`/`TV` still resolve via their base
  aliases).
  - Examples: `Sonarr 4K`→`sonarr`→`media`; `Radarr-Anime`→`radarr`→`media`;
    `Lidarr Music`→`lidarr`→`music`; `Readarr Audiobooks`→`readarr`→`book`;
    `Movies 4K`→`movies`→`media`.
- **Variant rendering is already supported:** `IconVariant` includes `"4k"` and
  `BuiltinIcon.variants` exists (reserved/unused) — so `builtin:<key>?v=4k` works today;
  only optional 4K-badged assets are missing.
- **Keep `suggestIconKey` pure (returns a bare key).** If 4K-badging is wanted, decorate
  `?v=4k` **one layer up** (picker/caller) when the original name contained `4k`/`uhd`
  and a `4k` variant asset exists — do **not** make the suggester emit query params.
- **Multi-instance** (`Sonarr`, `Sonarr 4K`, `Sonarr Anime`) → same base key;
  differentiate by the tile's app name (and optionally `?v=alt`), not by distinct icons.

## Trademark / brand risk assessment

Four tiers, lowest→highest risk:

1. **First-party generic glyphs (v0.2.6 default)** — we author/own; no copyright or
   trademark exposure. **Safest; keep as the default.**
2. **Third-party *generic* OSS packs (Lucide/Tabler/MDI-generic)** — permissive
   copyright, glyphs aren't marks; safe to bundle if license text is retained. Low risk.
3. **Official brand logos (selfh.st / dashboard-icons / simple-icons)** — **high
   trademark risk regardless of the permissive copyright license.** A *user* displaying a
   logo next to their own app instance is textbook **nominative fair use**; a *product
   redistributing a curated logo library* in its distributed artifact is the risky act
   (implied endorsement, brand-guideline violations, shipping marks for apps the user
   doesn't even run). → **user-import-only, never bundled.**
4. **User-uploaded icons** — liability shifts to the user; the product's duty is safe
   serving (raster-only, magic-byte sniff, `nosniff`/inline/immutable, opaque id).
   Already handled in v0.2.6; SVG stays blocked.

## Recommended safe icon policy

- **Bundle only first-party generic glyphs** (status quo). Optionally, generic OSS packs
  (Lucide/Tabler/MDI-generic) *may* be bundled with license text retained — but authoring
  first-party keeps a cohesive visual style, so this is optional, not a v0.2.7
  deliverable.
- **Never bundle brand logos.** Provide **user-driven** paths instead: existing custom
  upload (PNG/WebP) and a *future* opt-in icon-pack import; surface `license`/attribution
  + an "logos are property of their owners; identification only; no endorsement"
  disclaimer.
- **Keep SVG blocked** for uploads and packs until a reviewed server-side sanitiser + ADR
  land.
- **No remote fetch** without routing through the existing guarded-fetch SSRF chokepoint
  (`privateNetwork:"block"`).

## Recommended future icon-pack manifest shape (for the ADR; NOT built in v0.2.7)

- **Grammar:** add `pack:<packId>/<key>` (and `?v=<variant>`) alongside existing refs;
  falls through `parseIconRef` safely (back-compatible). Pack icons resolve to
  `mode:"img"` only (never mask).
- **Manifest (`manifest.json`, `manifestVersion:1`):** pack `id` (strict slug
  `^[a-z0-9][a-z0-9-]{0,62}$`), `name`, `version` (semver), `author`, `license` (SPDX),
  `homepage` (display-only), and `icons[]` with `{ key (slug), label, aliases[],
  file (assets/-rooted, no ..), mime (advisory), variants{light,dark,4k,alt}, license?,
  attribution? }`.

```json
{
  "manifestVersion": 1,
  "id": "homelab-core", "name": "Homelab Core", "version": "1.4.0",
  "author": "Jane Example", "license": "MIT", "homepage": "https://example.com",
  "icons": [
    { "key": "plex", "label": "Plex", "aliases": ["plex","media"],
      "file": "assets/plex.png", "mime": "image/png",
      "variants": { "dark": "assets/plex-dark.webp" } }
  ]
}
```

- **Validation:** archive size + entry-count + uncompressed (zip-bomb) caps; manifest zod
  schema; per-asset magic-byte sniff (PNG/WebP only, decided by bytes not declared
  `mime`); 512 KB/asset; sha256 dedup; path-traversal rejection; atomic staged import
  (temp → move on full success).
- **Storage:** `<ICONS_DIR>/packs/<packId>/` with assets stored by `<sha256>.<ext>` + an
  `icon_packs` table (start: one row + JSON icons). Backup = whole data dir (already
  documented).
- **Sources:** local `.zip` upload first (no SSRF surface); URL-fetch later **only**
  through `guardedFetch()` with `privateNetwork:"block"`.

## What should NOT be implemented yet

- ❌ Icon-pack import (zip/manifest), `icon_packs` table, `pack:` grammar — **future
  phase**, not v0.2.7.
- ❌ Bundling **any** third-party assets (brand *or* generic).
- ❌ User-uploaded SVG / SVG-in-packs (no sanitiser yet).
- ❌ Remote URL icon fetch / guarded-fetch import path.
- ❌ Auto-emitting `?v=4k` from `suggestIconKey`.
- ❌ Docker/deployment/auth changes; drag-and-drop; layout templates/import-export; any
  v0.3 work.

## Risks / open questions

1. **Curation/registry size** — 16→~27–30: confirm the new-key set (Tier 1 only, or
   +Tier 2?).
2. **`vpn` split vs keep-in-`shield`** — splitting requires removing `vpn`/`wireguard`
   from `shield`'s aliases (suggester tie-break favours earlier registry order).
   Recommend **keeping `vpn` in `shield`** for v0.2.7 to avoid alias surgery.
3. **Suggester scaling** — ~300+ aliases increases collision/tie surface; needs strong
   unit coverage; watch short-alias bleed.
4. **Ambiguous apps** — document canonical choices (Frigate→`camera` not `ai`;
   Tautulli→`chart` not `media`; Coolify/Dokploy→`container` not `code`; DMS→`notes`).
5. **Glyph design effort** — ~11–14 new first-party SVGs that read clearly as monochrome
   masks in light+dark.
6. **Bundling generic OSS packs** — allowed (Lucide/Tabler/MDI) but recommend authoring
   first-party for visual cohesion; decide explicitly.
7. **Scope split** — could ship alias-expansion-only first (zero new assets, lowest risk)
   and new keys second; or together. Recommend together for a coherent release.

## Acceptance criteria for v0.2.7

- New first-party generic keys added (Tier 1, optionally Tier 2), each with a monochrome
  SVG that renders via mask/currentColor and falls back to initials if absent.
- Alias expansions applied; `suggestIconKey` maps the catalogued **P1/P2** apps to the
  correct key (covered by unit tests), with deterministic tie-breaks and no <3-char
  cross-bleed.
- 4K/multi-instance pre-pass: `Sonarr 4K`→`media`, `Radarr-Anime`→`media`,
  `Lidarr Music`→`music`, bare `Anime`/`Movies`/`Photos` still resolve; never strips to
  empty.
- **No regression:** existing `builtin:`/`custom:`/URL/initials refs and existing apps'
  icons render identically; suggestion still never overwrites an explicit choice.
- **No DB migration** (`db:generate` reports no changes), **no apps-schema/Docker/auth
  changes**.
- ADR documenting the icon policy + future pack-import design is added.
- `pnpm typecheck`/`lint`/`test`/`build` all green.
