# v0.2.9 — Icon Pack App Matching (plan)

Planning basis for the v0.2.9 phase. Implemented on
`feature/v0.2.9-icon-pack-app-matching` from `main` at tag `v0.2.8.1`. See
`docs/adr/0016-icon-pack-app-matching.md` for the accepted decision and
`v0.2.9-implementation-report.md` for what shipped.

## Goal
After a user imports an icon pack, offer a review UI that **suggests** which pack
icons fit existing apps and **bulk-applies only the confirmed selection**. Icons
are **never auto-applied**: suggestion → review (tick/untick + override) →
confirmation dialog → apply.

## Architecture reused (no new primitives)
- `apps.icon` is a typed reference string; a pack assignment is just
  `apps.icon = pack:<packId>/<iconKey>` (ADR 0015). **No schema change.**
- `src/lib/icons/resolve.ts` — `buildPackRef`, `parseIconRef`.
- `src/lib/icons/suggest.ts` scoring → extracted to `match-core.ts` and shared.
- Docker import (`/apps/import`) is the template: pure logic + dedicated page +
  bulk POST + per-item outcomes.

## Approved decisions
1. Dedicated `/apps/icons` review page. 2. Native `<select>` override.
3. Pack-scoped route `POST /api/icons/packs/[packId]/apply`. 4. Partial success +
per-item outcomes. 5. No manifest aliases. 6. Exclude hidden/retired by default,
7. with an include toggle. 8. Existing-icon apps default unticked. 9. No-icon apps
default ticked at medium/high confidence. 10. Never auto-apply. 11. Always confirm.
12. Server enforces overwrite protection. 13. Primary entry from the Apps page.
14. In-picker post-import link is a secondary convenience. 15. Manual Packs-tab
selection unchanged.

## Matching algorithm
Pure `matchPackToApps(apps, pack)`. For each app: normalise + qualifier-strip the
name and URL host; score each pack icon's candidates (`key`, `label`, and their
stripped forms) with the shared `scoreAlias` (exact 3 / contains 2 /
reverse-contains 1, ≥3-char guard, longest-candidate tie-break). Best icon per app
with score > 0 becomes a suggestion. Qualifier stripping is bidirectional
(`sonarr` ↔ "Sonarr 4K", `sonarr-4k` ↔ "Sonarr").

- **confidence:** 3 → high, 2 → medium, 1 → low.
- **reason:** "Exact name match" / `App name contains "x"` / `URL address matches
  "x"` / `Icon "x" matches the app name`.
- **defaultSelected:** medium/high **and** the app has no existing icon.
- **currentIsCustomised / alreadySet:** computed against the suggested ref.

## UI (`/apps/icons`)
Pack selector · Include hidden/retired toggle (off) · Replace existing icons
toggle (off) · Select all / Clear · per-row [apply ✓] · app name/url · current →
proposed preview · confidence badge · reason · override `<select>` · Apply →
`ConfirmDialog` → result summary (applied/skipped/failed + per-app outcomes).
Responsive (wrapping rows, not a fixed table) for mobile + dark mode.

## API / server
`POST /api/icons/packs/[packId]/apply` (`protectedRoute`, POST). Body (zod):
`{ assignments: [{ appId, iconKey, variant? }], overwriteCustomised? }`. Service
`applyPackMatches` re-enforces everything (pack 404; unknown icon → skip; absent
variant → base fallback; app not found → skip; already-set → skip; existing icon →
skip unless `overwriteCustomised`). Partial success, per-item outcomes.

## Safety model
Three layers: review screen, confirmation dialog, server overwrite gate. The
client is never trusted; the server independently refuses to overwrite a custom
icon without the opt-in.

## Tests
- Unit `pack-match.test.ts`: match kinds (exact/label/host/substring/reverse),
  qualifier stripping both ways, no-match, confidence, reasons, default selection,
  customised/already-set detection, override ref. Plus a `suggestIconKey`
  regression block.
- Integration `pack-match.test.ts`: apply sets `apps.icon`; only selected updated;
  overwrite gate (off/on); already-set no-op; unknown icon skip; app-not-found
  skip; variant + fallback; mixed counts; pack 404; route CSRF (403 / same-origin
  200 / 404).

## Out of scope
No schema/migration, no new deps, no manifest aliases, no remote imports, no
bundled packs/logos, no SVG, no pack replace/upgrade, no Docker/auth/deployment
change, no v0.3 work.
