## Existing app/widget architecture summary

**The dashboard is a versioned, catalog-resolved layout document** (ADR 0011):

- **Persisted document** `DashboardLayoutConfig` = `{ version, sections[] }`; each section `{ id, title, order, widgets[] }`; each `PlacedWidget` `{ id, widgetKey, hidden, size, order, config? }`. Only stable keys are stored — `config` is reserved and currently unused (`src/lib/types/widgets.ts`).
- **Widget catalog** (`src/server/dashboard/catalog.ts`) is built from enabled **modules'** `WidgetDefinition`s (`src/modules/core/index.ts` registers the 4 built-ins). It is the single source of truth for "known" widgets. `widgetKey == WidgetDefinition.id`; `componentKey` binds to a client component.
- **Reconcile** (`src/server/dashboard/reconcile.ts`): drops widgets whose `widgetKey` is **not in the catalog**, drops duplicate instance `id`s (keep first), **auto-appends any catalog widget missing everywhere to the first section, visible**, and renumbers order.
- **Resolve** (`resolveLayout`) joins catalog metadata (`title`, `componentKey`, `columns`) into a render-ready `DashboardLayoutDTO`. Pure — **no DB**.
- **Default layout** (`src/server/dashboard/default-layout.ts`): a canonical 4-widget arrangement; **any other catalog widget is auto-appended**.
- **Persistence/service** (`src/server/services/dashboard-layout.ts`): one global row (`kind='user-default'`, `ownerKey=NULL`); reads are crash-proof (fall back to computed default); `GET`/`PUT`/`POST reset` routes.
- **Client**: `dashboard-grid.tsx` renders each widget inside a `Card` (title from `ResolvedWidget.title`), body via `getWidgetComponent(componentKey)` rendered as **`<Widget />` with no props**. `widget-registry.tsx` maps componentKey → component. Widgets (e.g. `favourite-apps-widget.tsx`) **self-fetch** the full apps list via `fetchApps` and render name/icon/health/Open.
- **Editor** (`dashboard-editor.tsx` + pure `dashboard-editor-logic.ts`): draft is a `DashboardLayoutDTO`; pure transforms for show/hide, resize, reorder, move-between-sections, add/rename/delete sections; `dtoToConfig` strips render-only fields and renumbers; saves whole document via `PUT`.
- **Apps model**: `apps` table + `AppDTO` (`src/lib/types/apps.ts`) with `name, url, icon, categoryId, isHidden, lifecycle (active|retired), latestHealth`, etc. `GET /api/apps` (list, filterable) and `GET /api/apps/:id` (single, 404 if absent) both exist. `CategoryDTO` provides category names. No SWR/react-query — plain `fetch` wrappers.
- **Ownership** (`docs/agents/ownership.md`): `src/lib/types/**`, `src/lib/validation/**`, `src/modules/types.ts`, `src/db/schema/**` are ⭐ **Architect-owned**; `src/server/**` + `src/app/api/**` Backend; `src/components/**` + `src/hooks/**` + `src/app/(dashboard)/**` Frontend; `tests/**` QA.

**Critical implication:** any "app widget" must (a) have a `widgetKey` the catalog recognises or reconcile will **silently delete it**, and (b) **not** be subject to the catalog auto-append rule, or every app/the template would be force-added to the dashboard.

## Proposed architecture

**Decision (planning step 9): Option A — one generic app-widget type with the app id in `config`.** Strongly recommended over per-app generated registry entries.

| | **A. Generic widget + `config.appId`** ✅ | **B. Generated catalog/registry entry per app** |
|---|---|---|
| Catalog purity | Stays module-derived & pure | Becomes DB-dependent (impure) |
| Auto-append rule | One generic entry, flagged non-auto-place → no apps auto-added | Every app becomes a catalog entry → **reconcile/default would auto-add every app** (violates "don't add every app") |
| Client registry | One `"app"` component, keyed by `config.appId` | Still needs one generic component anyway (can't static-map per-app) |
| Duplicate/missing apps | Clean: id `app:<id>`, render-time unavailable state | Awkward: removing an app must mutate the catalog |
| Tests (`widget-registry.test.ts` asserts every catalog componentKey maps to a component) | Holds | Breaks unless all map to the generic component |

Option B offers no real benefit and fights every existing invariant. **Go with A.**

To make A safe, introduce an **`instanceable`** concept across the contract: a catalog entry that is a *template* the user instantiates 0..N times, which the framework must **never auto-place** and which the editor treats as **addable/removable**.

- Persisted instance: `{ id: "app:42", widgetKey: "app", config: { appId: 42 }, size, hidden, order }`.
- The generic `"app"` `WidgetDefinition` lives on the **core** module, flagged `instanceable: true`.
- Reconcile keeps instances (widgetKey `"app"` is "known"), dedups by id, and **skips instanceable entries in the auto-append loop**.
- Default layout **skips instanceable entries** → fresh/reset dashboards contain no app widgets.
- Missing/deleted apps are a **render-time** concern (unavailable state), never a reconcile-time deletion.

## Layout config representation
```jsonc
{
  "id": "app:42",          // app:<appId> → enforces one widget per app; dedup-by-id backstop
  "widgetKey": "app",      // generic instanceable catalog key (core module)
  "hidden": false,
  "size": "small",
  "order": 3,
  "config": { "appId": 42 } // the ONLY per-instance state persisted; name resolved fresh each load
}
```
Only `appId` is persisted (no denormalised name) so the title never goes stale; the live name is resolved server-side at load and the editor shows the picked app's current name.

## Server-side resolution
1. `buildWidgetCatalog()` includes the `instanceable` `"app"` entry.
2. `reconcileConfig`: instances survive (`"app"` is known) and dedup by id; the `"app"` template is **excluded from auto-append**; instances with an invalid `config.appId` are dropped (defensive only).
3. `resolveLayout`: joins generic `componentKey "app"`, `instanceable`, generic title.
4. `dashboard-layout.ts` post-resolve enrichment: replace each app widget's `title` with the app's current name (or "Unavailable app" if not found) via a single apps lookup. Live health/category still fetched client-side.

## Client rendering
`AppWidget({ config })`:
- Resolve `appId`; fetch the app (`fetchApp(id)`, or shared cache).
- **Found:** icon/initial, name, health badge (reusing `healthTone`/`healthLabel` only when `healthEnabled`), category badge where available, **Open** (honouring `openNewTab`).
- **Missing/deleted:** `EmptyState`/muted "This app is no longer available — remove it in Customise."
- **Retired:** muted "Retired" treatment; Open shown only if a URL is known.
- Never throws; mirrors the loading/error/empty pattern of the existing widgets.

## Editor add-widget flow
- An **"Add app widget"** control opens `add-app-widget.tsx`: a Select of apps (from `fetchApps`, active) + a target-section Select (default = last section). Apps already placed are disabled ("Already added").
- Confirm → `addAppWidget(draft, app, sectionId)` appends a `ResolvedWidget` (title from the picked app, `componentKey:"app"`, `widgetKey:"app"`, `config:{appId}`, default size, `instanceable:true`), marks the draft dirty.
- Save persists via the existing `PUT` — **no API change**.

## Duplicate handling
One widget per app: id `app:<appId>`; `addAppWidget` no-ops if the id exists; the picker disables placed apps; `reconcileConfig` dedup-by-id is the server backstop. (Multiple instances per app is a possible later extension via suffixed ids — **not** in v0.2.4.)

## Missing/deleted/retired app handling
- **Never auto-removed** server-side (a transient/async miss must not delete the user's widget). Reconcile drops an instance only if `config.appId` is structurally invalid.
- Rendered as an **Unavailable/Retired** state; the editor's **Remove** lets the user clean it up.
- Title falls back to "Unavailable app" in the resolved DTO so the card header is sane even before client fetch.

## Reset/default behaviour
Unchanged and safe: `resetLayout` clears the row → `buildDefaultLayout` (which skips instanceable) → built-in 4-widget default with **no app widgets**. The existing Reset confirm dialog already warns this removes customisations. This matches the brief; no better behaviour proposed (per-user layout retention is out of scope).

## Validation/reconciliation changes
- `instanceable` excluded from auto-append in **both** `reconcile.ts` and `default-layout.ts`.
- Instances kept by widgetKey; dedup by id unchanged.
- *(Recommended)* `placedWidgetSchema` requires `config.appId` (positive int) for `widgetKey "app"`; reconcile drops malformed instances defensively.
- `ResolvedWidget`/`WidgetCatalogEntry` carry `instanceable`.

## How this avoids breaking existing built-in widgets
- The 4 singletons keep `instanceable` falsy → unchanged auto-append, default arrangement, and reconcile behaviour.
- Grid passes props additively; prop-less components ignore them (type-safe).
- `resolveLayout` stays pure (enrichment is a separate service pass) → existing `dashboard-layout.test.ts` unaffected.
- No API/route/DB-schema/migration change; existing saved layouts load identically.

## Risks/open questions
1. **Card-title resolution needs a DB read** in the load path — proposed as a service-layer enrichment to keep `resolveLayout` pure. Confirm the split vs. passing an app map into `resolveLayout`.
2. **N fetches** if each AppWidget self-fetches — recommend the shared `use-apps-cache` hook.
3. **Picker scope:** show only active, non-hidden apps, or also allow pinning hidden/retired apps deliberately? (Default proposal: active apps, including hidden, excluding retired.)
4. **One-per-app vs multiples** — proposal disallows multiples in v0.2.4.
5. **Remove vs Hide** for app widgets — proposal adds Remove (instanceable only).
6. **Open on retired/deleted** apps when no URL is known — proposal hides Open.
7. **Validation strictness** — tighten schema for `config.appId`, or rely solely on reconcile? (Proposal: both.)

## Acceptance criteria
- A user can add an existing app as a standalone homepage widget from the editor; it persists across reloads.
- App widgets support show/hide, resize, reorder, move-between-sections, save/cancel/reset, **and remove**.
- App widget shows name, status (where available), category badge (where available), icon/initial, and Open.
- Deleted/retired/missing apps render an unavailable state and **never crash** the dashboard.
- Default and Reset layouts contain **no** app widgets; no app is ever auto-added.
- All 4 built-in widgets render and behave exactly as before; existing saved layouts load unchanged.
- No API contract, DB schema, or migration changes.

## Recommendation
Proceed with **Option A (generic `"app"` widget + `instanceable` + `config.appId`)** on branch `feature/v0.2.4-individual-app-widgets`. It fits the existing catalog/reconcile/resolve model with **no API, schema, or migration changes**, keeps the default/reset dashboard safe, and isolates missing-app failures to render time. The remaining judgement calls are the open questions above (esp. #1 title-resolution split, #2 shared fetch, #3 picker scope).
