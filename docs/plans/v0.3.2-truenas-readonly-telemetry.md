# v0.3.2 ‚Äî TrueNAS Read-only Telemetry Provider

Adds an **optional, read-only** TrueNAS telemetry provider behind the existing
v0.3.0 infrastructure telemetry abstraction. It enriches the Infrastructure page
with real NAS pool, dataset, disk, temperature and SMART-style health data **where
available**, while keeping app/container storage clearly separate and the page
fully usable when TrueNAS is not configured.

This is **read-only**. No write actions, pool/dataset management, SMART test or
scrub triggering, shell execution, or destructive operations of any kind.

## Design decisions

### Transport ‚Äî hand-rolled WebSocket over `node:tls`
TrueNAS SCALE's current API is **JSON-RPC 2.0 over a WebSocket** at `/api/current`
(the legacy `/api/v2.0` REST routes are deprecated and version-fragile, per the
brief). We speak that protocol directly with a small dependency-free client ‚Äî
matching the project's existing "speak the protocol directly, no third-party
library" ethos (cf. the Docker engine client over `node:http`).

Why hand-rolled rather than Node's global `WebSocket`: the global (undici)
`WebSocket` cannot be given per-connection TLS options (custom CA / verification
bypass) without adding the `undici` dependency. `node:tls.connect` gives native
per-connection TLS control, so we can verify by default and scope any opt-in
bypass to **only** the TrueNAS connection.

### TLS rules (operator-confirmed)
- Certificates are **verified by default** (`rejectUnauthorized: true`).
- `TRUENAS_ALLOW_INSECURE=true` is the **only** way to skip verification; unset ‚Üí
  treated as `false`.
- Insecure mode affects **only** the TrueNAS connection ‚Äî never global TLS. We do
  **not** use `NODE_TLS_REJECT_UNAUTHORIZED=0` and do **not** add `undici`.
- Insecure mode is documented as being for trusted local / self-signed homelab
  deployments only. With a valid cert (or a TLS reverse proxy) keep it `false`.

### Module layout (`src/server/telemetry/truenas/`, server-only)
- `types.ts` ‚Äî strict raw (TrueNAS) + provider result types.
- `ws-client.ts` ‚Äî hand-rolled WebSocket client over `node:tls`/`node:net` (frame
  encode/decode, masking, handshake, ping/pong, timeouts). Transport only ‚Äî thin,
  not unit-tested (like the Docker `engine.ts`).
- `client.ts` ‚Äî JSON-RPC 2.0 client: `auth.login_with_api_key`, concurrent
  `call(method, params)` with per-call timeouts.
- `normalise.ts` ‚Äî **pure** normalisers: raw pools/datasets/disks/temps/SMART ‚Üí
  DTOs, severity mapping, capacity parsing, serial masking. Heavily unit-tested.
- `provider.ts` ‚Äî orchestrator: reads env, connects, queries (best-effort,
  bounded by an overall budget), normalises, returns a calm result. **Never
  throws**; never logs secrets; never returns secrets/host paths/raw errors.

### DTO integration
`StorageTelemetryDTO` gains `nas: NasTelemetryDTO | null`:
- `null` when TrueNAS is **not configured** (page renders exactly as before).
- otherwise an object carrying `status` (available/partial/unavailable), a
  severity rollup, NAS `pools`, `datasets`, and `disks` (reusing `DriveDTO`).

The existing `storage.pools`/`storage.drives` (container-visible local
filesystems / `/sys` drives) are unchanged, so NAS and app/container storage stay
clearly separated. The `truenas` entry in the `sources` array is now driven by
the real provider result instead of the hard-coded `not_configured` placeholder.

### Source status mapping
- env missing ‚Üí `not_configured`.
- unreachable / TLS / protocol failure ‚Üí `unavailable` (calm).
- auth failure ‚Üí `unavailable` with an "authentication failed ‚Äî check the API key"
  detail (no new enum value; no raw error).
- connected + core query ok, some optional query failed ‚Üí `partial`.
- everything ok ‚Üí `available`.

### Severity / privacy rules
- Pool `ONLINE`/healthy ‚Üí healthy; `DEGRADED`/unhealthy ‚Üí warning; `FAULTED`/
  `UNAVAIL`/`REMOVED` ‚Üí critical; unknown ‚Üí unavailable.
- SMART `PASSED`/`SUCCESS` ‚Üí passed; `FAILED`/`FAILING` ‚Üí failing; else
  unavailable. Missing telemetry is **unavailable, never critical**.
- Temperature reuses the existing bands (<45 / 45‚Äì54 / ‚â•55 ¬∞C).
- Disk serials are masked via the existing `redactSerial` (full serials never
  reach the client). Dataset **names** (`pool/child`) are surfaced, never host
  mountpoints.

## Out of scope (unchanged)
No write actions, pool/dataset/disk management, SMART test or scrub triggers,
alert acknowledgement, shell commands, host filesystem browsing, Docker/Dockerfile
changes, data-volume permission fix, schema changes, auth/multi-user, AI, notes,
or dashboard customisation. No merge, no tag, no next phase.

## Env vars
- `TRUENAS_URL` ‚Äî TrueNAS Web UI/API URL reachable from the dashboard server
  (e.g. `https://truenas.example.local`). Optional.
- `TRUENAS_API_KEY` ‚Äî read-only/least-privilege API key. Server-side only.
- `TRUENAS_API_PATH` ‚Äî JSON-RPC WebSocket path, default `/api/current`.
- `TRUENAS_ALLOW_INSECURE` ‚Äî `true` to skip TLS verification (self-signed homelab
  only); default `false`.

## Tests
Parser/normaliser/provider-helper level with mocked TrueNAS responses: pool /
dataset / disk / temperature / SMART normalisation, capacity parsing
(`{parsed}` and raw numbers), severity mapping, serial masking, config detection,
status mapping, and defensive handling of missing/garbage fields. Transport
framing is not unit-tested (best-effort, degrades to `unavailable`).
